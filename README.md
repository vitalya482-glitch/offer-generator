# SAM Offer Generator

Генератор коммерческих предложений из Excel-калькуляции и Word-шаблона.

Проект нужен для быстрого формирования КП по выбранному направлению/бренду. Пользователь выбирает папку проекта, Excel-калькуляцию, Word-шаблон и лист Excel. Программа читает данные из Excel, подставляет их в DOCX-шаблон и сохраняет готовое КП.

Главная логика производительности сейчас находится в:

```text

core/excel_reader.py      # чтение Excel + кеш + индексация листа

```

Самые важные идеи оптимизации:

1. Не сканировать проектную папку на каждый ввод символа.
2. Кешировать результат сканирования папки.
3. Не читать Excel-лист много раз.
4. Один раз индексировать Excel-лист и дальше искать по индексу.
5. Открывать Excel через `read_only=True, data_only=True`.
6. По возможности не выполнять тяжёлые операции в UI-thread.

## Основной сценарий работы

1. Пользователь выбирает папку проекта на сервере или локальном диске.
2. Программа сканирует папку и находит:
   - Excel-файлы: `.xlsx`, `.xlsm`
   - Word-шаблоны: `.docx`
   - PDF-папки, если они нужны для дальнейшей логики
3. Пользователь выбирает направление:
   - Stulz
   - Riello
   - DC Eltek
   - Generator
4. Пользователь выбирает Excel, Word-шаблон и лист Excel.
5. Программа парсит Excel.
6. Программа формирует Word КП в выбранную папку результата.

# Логика производительности

## 1. Сканирование папки проекта

Раньше папка проекта могла пересканироваться слишком часто, особенно при изменении текста в поле пути. Это сильно тормозило работу, особенно если проект лежит на сервере или содержит много файлов.

Правильная логика:

- не запускать полный scan на каждый символ;
- использовать debounce через `QTimer`;
- запускать scan только после паузы ввода или после выбора папки;
- кешировать результат scan для уже выбранной папки;
- при повторном выборе той же папки брать данные из кеша.

Типовая идея:

```python
self.scan_timer = QTimer()
self.scan_timer.setSingleShot(True)
self.scan_timer.timeout.connect(self._scan_project)

self.project_edit.textChanged.connect(
    lambda: self.scan_timer.start(700)
)
```

Также важно исключать тяжёлые и ненужные папки:

```python
SKIP_DIRS = {
    "__pycache__",
    ".git",
    "backup",
    "archive",
    "old",
    "temp",
}
```

Рекомендуемый обход папок:

```python
for root, dirs, files in os.walk(project_dir):
    dirs[:] = [d for d in dirs if d not in SKIP_DIRS]
```

Это быстрее и лучше контролируется, чем простой `Path.rglob("*")`.

---

## 2. Кеширование сканирования проекта

Кеш должен хранить результат для конкретной папки проекта.

Простая логика ключа:

```python
cache_key = str(project_dir.resolve())
```

Что можно хранить:

- список Excel-файлов;
- список Word-шаблонов;
- список PDF-папок;
- время последнего сканирования;
- признак, что папка уже была просканирована.

Базовая логика:

```python
if cache_key in self._scan_cache:
    return self._scan_cache[cache_key]

result = scan_project_files(project_dir)
self._scan_cache[cache_key] = result
return result
```

Если нужно сделать кеш точнее, можно добавить invalidation по `mtime` корневой папки или кнопку «Обновить».

Практичный вариант для GUI:

- по умолчанию использовать кеш;
- добавить кнопку «Пересканировать»;
- при нажатии очищать кеш только для текущей папки.

## 3. Excel: почему было медленно

Главное узкое место — многократное полное сканирование одного и того же Excel-листа.

Типичная медленная логика:

```python
for row in ws.iter_rows():
    for cell in row:
        ...
```

Если такая логика вызывается много раз для поиска разных label-ов (`TOTAL`, `VAT`, `RATE`, `MODEL`, `QTY` и т.д.), один и тот же лист проходится 10–20 раз.

`openpyxl` сам по себе не очень быстрый, поэтому повторные проходы по листу резко замедляют проект.

---

## 4. Excel indexing: правильная логика

Нужно один раз пройти по листу и построить индекс.

Идея:

```python
label_index = {}

for row in ws.iter_rows():
    for cell in row:
        value = normalize(cell.value)
        if value:
            label_index.setdefault(value, []).append((cell.row, cell.column))
```

После этого поиск label-а должен идти не через новый полный scan, а через индекс:

```python
cells = label_index.get("total", [])
```

Для частичного поиска можно хранить также плоский список текстовых ячеек:

```python
text_cells = [
    (row, col, normalized_value),
    ...
]
```

Тогда функции типа `first row containing` работают по уже подготовленным данным, а не читают Excel заново.

## 5. Кеширование Excel

Кешировать можно на нескольких уровнях.

### 5.1 Workbook cache

Ключ:

```python
(path, mtime)
```

Где:

```python
mtime = os.path.getmtime(path)
```

Если файл не изменился, можно повторно использовать уже прочитанные данные.

### 5.2 Sheet index cache

Ключ:

```python
(path, sheet_name, mtime)
```

Хранить:

- значения листа;
- label index;
- text cells;
- max row;
- max column;
- найденные model groups;
- результат парсинга, если он не зависит от UI-параметров.

### 5.3 Parsed offer cache

Ключ:

```python
(path, sheet_name, brand, mtime)
```

Хранить результат парсинга Excel в структуру данных КП.

Важно: если пользователь меняет параметры, которые влияют на итоговое КП, не нужно переиспользовать неподходящий parsed result.

## 6. Как открывать Excel

Для чтения использовать:

```python
load_workbook(path, read_only=True, data_only=True)
```

Плюсы:

- быстрее открытие;
- меньше расход памяти;
- читаются рассчитанные значения формул;
- не нужно загружать лишние стили и объекты.

Минусы:

- нельзя редактировать workbook;
- некоторые операции с layout/merged cells могут быть ограничены.

Для текущей задачи чтения КП это обычно правильный режим.

---

## 7. Что ещё можно кешировать

### Список листов Excel

При выборе Excel программа часто получает список листов. Это можно кешировать отдельно.

Ключ:

```python
(path, mtime)
```

Значение:

```python
workbook.sheetnames
```

### Найденные model groups

`parse_model_groups()` может быть дорогим, особенно на широких Excel. Его результат можно хранить в sheet index cache.

### Нормализованные значения ячеек

Функция normalize часто вызывается много раз. Лучше нормализовать значения один раз при построении индекса.

### Результат поиска label-ов

Если есть функции вида:

```python
find_row_by_label("TOTAL")
find_cells_by_label("VAT")
```

их результаты можно хранить в dict внутри объекта индекса.

### Word template structure

Если один и тот же DOCX-шаблон используется много раз, можно кешировать некоторые сведения:

- наличие таблиц;
- позиции тегов;
- список placeholder-ов.

Но сам `Document()` из `python-docx` лучше каждый раз открывать заново, чтобы не получить побочные изменения между генерациями.

## 8. GUI и зависания

Даже после кеширования тяжёлые операции могут занимать время:

- сканирование большой сетевой папки;
- открытие большого Excel;
- генерация DOCX;
- копирование/сохранение файла.

Чтобы интерфейс не зависал, эти операции лучше вынести из UI-thread:

- `QThread`
- `QRunnable`
- `QThreadPool`

Что желательно вынести в background:

```text
scan_project_files()
load Excel sheets
parse Excel
render DOCX
```

Минимально достаточно сначала вынести генерацию КП и сканирование проекта.

## 9. DOCX renderer

`python-docx` может быть медленным на больших шаблонах и таблицах.

Потенциальные узкие места:

```python
for table in doc.tables:
    for row in table.rows:
        for cell in row.cells:
```

и:

```python
deepcopy(source_row._tr)
```

Что можно улучшить позже:

- объединить несколько проходов по документу в один;
- не искать теги повторно;
- заранее строить карту placeholder-ов;
- для больших шаблонов рассмотреть `docxtpl`.

Пока основной bottleneck — Excel и scan проекта, поэтому DOCX оптимизировать вторым этапом.

# Важные правила для будущих изменений

1. Не добавлять новые полные проходы по Excel без необходимости.
2. Если нужна новая функция поиска по Excel, сначала проверить, можно ли взять данные из индекса.
3. Любой кеш должен иметь понятный ключ и invalidation.
4. Для файлов кешировать с учётом `mtime`.
5. GUI не должен запускать тяжёлые операции напрямую при каждом изменении поля.
6. Если проектная папка на сервере, любые лишние `rglob`, `exists`, `stat` и открытие файлов будут заметно тормозить.
7. Не кешировать изменяемый объект Word `Document` между генерациями.
8. Если Excel-файл открыт пользователем и не пересчитан, `data_only=True` может вернуть старые рассчитанные значения формул. Это нормальное поведение openpyxl.

# Где смотреть при проблемах

## Медленно выбирается папка проекта

Смотреть:

```text
core/project_scanner.py
gui/main_window.py
```

Проверить:

- не запускается ли scan повторно;
- работает ли debounce;
- используется ли кеш;
- не сканируются ли ненужные папки.

## Медленно выбирается Excel или лист

Смотреть:

```text
core/excel_reader.py
gui/main_window.py
```

Проверить:

- кешируется ли список листов;
- не открывается ли workbook повторно;
- используется ли `read_only=True`.

## Медленно формируется КП

Смотреть:

```text
brands/stulz.py
core/excel_reader.py
core/docx_renderer.py
```

Проверить:

- не парсится ли Excel заново;
- не выполняются ли повторные поиски label-ов через полный scan;
- не слишком ли большая таблица DOCX;
- не блокируется ли UI-thread.

# Важно

Шаблоны КП и калькуляции не хранятся внутри проекта. Программа работает с файлами, которые пользователь выбирает из папки проекта.

Для ускорения главное не добавлять повторное чтение файлов и повторные полные проходы по Excel/папкам. Любая новая логика должна сначала пытаться использовать уже построенный индекс или кеш.
